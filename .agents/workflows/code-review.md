# 嵌入式固件代码审查与修改工作流

> 基于 Eric820 项目审查经验总结，适用于所有嵌入式固件的代码审查和修改任务。

## 一、审查前准备

1. **读取产品文档**：查找项目目录中的说明书、技术规范、协议文档（PDF/Word），用 pypdf 提取文本
2. **建立工程地图**：列出所有工程、源文件、线程模型、中断、通信接口
3. **确认芯片与工具链**：STM32 / HC32 / 其他，Keil / GCC，避免混用 HAL API

## 二、代码阅读铁律

### 规则 1：三段式验证法（每个关键判断必做）

```
① 读源头 → grep -rn '变量名' 所有相关文件 → 找到定义处
② 读调用 → 找到所有赋值和引用处
③ 读上下文 → 确认注释符号(//)、条件编译(#if)、被注释掉的代码
```

// turbo-all
```bash
grep -rn '目标变量或函数' 工程目录/ --include='*.c' --include='*.h'
```

### 规则 2：注释敏感

- 必须检查目标行是否被 `//` 或 `/* */` 注释
- 被注释的代码可能曾经是有效逻辑，需要确认当前替代实现在哪里

### 规则 3：变量追踪到赋值处

```
❌ 看到函数参数名就猜测来源
✅ grep 变量名 → 找到赋值语句 → 确认计算公式和更新频率
```

### 规则 4：不用默认值描述可配参数

```
❌ "U0 > 65V"
✅ "U0 > settings.u0_limt（默认65，可配）"
```

### 规则 5：业务逻辑判断必须参考产品文档

- 遇到不确定的业务逻辑 → 先查说明书
- 说明书不够 → 直接问用户确认，不猜测
- 电力保护、工业控制等专业领域必须谨慎

### 规则 6：通信底层修改必抓"全链路一致"

涉及 CAN、UART、SPI 等通信底层参数（波特率、采样点、时序容差 SJW、帧格式等）的排查与修改时，**严禁只修改或确认单端设备**。必须打开全量配套通信硬件的底层配置文件，强制执行"发送端 → 中继端 → 接收端"三端参数完全对齐的严格排查。

```
必查参数清单（CAN 示例）：
  ① 波特率（Baud Rate）— 三端计算值必须一致
  ② 采样点（Sample Point %）— 三端偏差不超过 5%
  ③ 重同步跳转宽度（SJW）— 与线路环境匹配
  ④ 每位时间量子数（TQ/bit）— 影响抗干扰分辨率
  ⑤ 帧类型（标准帧/扩展帧）— 全链路统一
  ⑥ 过滤器 ID/Mask — 确保目标帧能通过
```

**教训来源**：HC32 采集器采样点被从 80% 改为 62.5%，而主机保持 80%、光纤模块 66.7%，三端不一致导致现场多节点通讯失败。修改时仅看了 HC32 一端代码，未核对主机和光纤模块。

## 三、修改代码前检查清单

### 修改前

1. `grep -rn '被修改的函数/变量' 所有工程目录` — 确认影响范围
2. 确认目标工程和芯片类型（STM32 vs HC32）
3. 确认当前编译器（Keil ARMCC 还是 GCC），使用正确的语法
4. 检查是否涉及中断上下文、DMA、volatile

### 修改后

1. 检查是否引入新的编译器私有语法
2. 确认条件编译兼容性（`#if defined(__ARMCC_VERSION)` / `#elif defined(__GNUC__)`）
3. 所有新增代码附带中文注释

## 四、输出前自检清单

每次给出分析结论前，过一遍：

- [ ] 被注释的代码行是否遗漏？
- [ ] 变量追踪到定义和赋值了吗？
- [ ] 阈值/参数是硬编码还是可配？
- [ ] 结合产品说明书/领域知识验证了吗？
- [ ] 有没有同名但不同作用域的变量？
- [ ] 函数调用链完整吗（是否只看了入口没看函数体）？
- [ ] 默认值和实际值区分了吗？

## 五、常见误判模式（红线）

| 误判模式 | 防范措施 |
|---|---|
| 被注释代码当作有效代码 | 全文搜索时注意 `//` 前缀 |
| 相似变量名混淆（sud_ua vs stable_ua） | 必须 grep 到赋值处 |
| 默认值当固定值 | 标注"可配"并追踪 settings 来源 |
| 用软件思维解读电力保护逻辑 | 先查说明书再下结论 |
| 只看函数调用不看函数体 | 每个关键函数必须读完整体 |
| && 和 \|\| 逻辑错误漏检 | 特别关注多条件组合中的互斥条件 |

## 六、Git 操作规范

- `origin` (Gitee) → **仅拉取**，禁止推送
- `github` (GitHub) → **唯一推送源**
- 每次修改前确认当前分支和最新 commit
- 推送前 `git diff --stat` 确认改动文件列表

## 七、多端协同编译与固件交付规范

### 规则 7：固件交付必须是可直接烧录的二进制产物

项目交付给测试人员的固件必须是 **.hex**（推荐）或 **.bin** 格式的可直接烧录文件。**严禁抛掷源码要求测试端自行编译**，防范因构建环境（Keil vs GCC）、库版本、操作系统（macOS vs Windows）不同带来的二次错误。

```
固件交付文件命名规范：
  <板名>_<版本号>_<git短hash>_<日期>.hex
  示例：canbox_v1.2.3_a1b2c3d_20260410.hex
```

### 规则 8：异构老工程必须建立命令行编译能力

所有工程（含 Keil 专属的老项目）必须具备基于命令行交叉编译工具（`arm-none-eabi-gcc` 等）一键无 GUI 输出烧录件的能力。为非 scons/Makefile 的异构工程手建轻量编译脚本（Makefile 或 CMakeLists.txt），保证全线产品在 CI/CD 环境和任意开发者电脑上都能无障碍构建。

**教训来源**：光纤转 CAN 模块（STM32F103）为纯 Keil 工程，macOS 开发者无法编译验证，修改后只能传源码给 Windows 同事重编，流程断裂且易出错。

### 规则 9：CI/CD 为固件交付唯一出口

生产级固件必须通过 CI/CD 流水线（Gitea Actions / Jenkins 等）统一编译产出，禁止从开发者个人电脑直接拷贝 .hex 给测试。CI 产物需附带：
- Git commit hash（可溯源）
- 编译工具链版本（可复现）
- 自动化构建日志（可审计）
